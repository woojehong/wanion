## 1. 비전과 목적

- **표면 목적**: 흩어진 와우 공동체 운영 도구(웹·디스코드·인게임·로그)를 하나의 플랫폼으로 통합
- **전략 목적**: '후제' 브랜드 구축 — 좋은 것을 계속 내놓는 사람으로서의 반복 노출. 플랫폼명은 중립(REKO), 전 페이지 "Made by 후제공방" 크레딧, **업데이트 노트(개발일지)를 정식 메뉴**로 운영
- **동반 홍보 구조**: 한길련(연합 페이지) · 각 길드(길드 페이지) · TeamSAD(공대 페이지)가 각자의 공개 프로필로 홍보되는 삼중 깔때기
- 포지셔닝: 랠리포인트 = 모르는 사람 **매칭** / REKO = 아는 사람들 **운영** (경쟁이 아니라 하류 공정)

## 2. 세계관 — 4엔티티 + 무소속 공대장

한국 와우의 현실: **길드(생활·소속)와 공격대/정공(운영·전투)은 다른 축.** 이 구조를 담은 최초의 플랫폼.

- **길드** — 소속의 단위. **등록제**: 창립 게이트 7슬롯 (와우팩토리 · 교차로 · 이에요 · 스타폴 · 두부킴의유기견들 + 예정 2). 미등록 길드는 플랫폼 세계관에서 미존재 → 등록의 희소가치가 홍보 동력. 추후 신청제 → 오픈
- **공격대(정공)** — 운영의 단위. 여러 길드 사람들로 구성, 길드와 독립. TeamSAD가 대표
- **연합** — 길드들의 묶음. 한길련이 창립 연합 ("이름 포기"가 아니라 창립 연합으로 격상)
- **유저** — 계정 1개, 복수 소속 기본
- **무소속 공대장** ★ — 소속이 없어도 누구나 파티(이벤트)를 개설하고 전국 단위로 모집 가능. 권한 모델상 "이벤트 스코프 호스트 역할". 프로필에 `무소속 · 공대장` 뱃지
- 모든 길드·공대·연합은 **비로그인 공개 프로필 페이지** 보유: 소개 · 모집 상태 · 프로그레스(자동 갱신) · [가입 신청]

## 4. 스케줄 발견성 — 같은 데이터, 3개의 시선

> "여러 레이드 스케줄이 어떤 게 있는지 유저들이 잘 볼 수 있게."

1. **탐색 (보드)** — 모집 상태 중심 리스트. 필터: 주최(연합/길드/공대/개인) · 난이도 · **역할 빈자리(킬러 필터: "힐러 자리 있는 공대만")** · 날짜. 정렬: 모집 중 우선 → 마감 임박순. 슬롯 스퀘어 시각화
2. **달력** — kgusystem 달력 계승 + 새 디자인. 주간 스트립(상시) + 월간(전환)
3. **내 일정** — 내가 신청/확정된 것만. C안(2패널) 구조로 관리까지
- 복수 공대 소속이 기본이므로 **일정 충돌 감지**("그 시간 TeamSAD 공대 있어요") 제공

## 5. 권한 모델 — memberships = (스코프, 역할) 집합

원칙 4: ① 권한은 계정 등급이 아니라 멤버십 레코드 여러 줄 ② 합집합(높은 권한 우선, 충돌 없음) ③ 스코프 격리(길드 관리자 ≠ 공대 권한) ④ 파생 소속(연합 소속 = 길드 경유 자동 계산).

| 스코프 | 역할 |
|---|---|
| 플랫폼 | 운영자(후제) · 스태프(추후) |
| 연합 | 운영진 · (파생)구성원 |
| 길드 | 마스터(1) · 관리자(N) · 길드원 |
| 공격대 | 공대장 · 운영진 · 공대원 · 용병(게스트) |
| 이벤트(개별 공대) | 호스트(=개설자, 무소속 가능) |
| 무소속 | 열람 · 가입 신청 · 용병 지원 · 파티 개설 |

엣지 확정: 복수 길드 가입 허용(본캐/부캐) + 대표 소속 1개 표시 · 마스터는 길드당 1인 · 밴은 별도 레이어 · 탈퇴 시 기록 보존.

## 6. 계정 & Battle.net 검증 플로우 (기술 검증 완료)

### 6.1 로그인/연동
Google(간편 로그인) + **Discord(필수)** + Battle.net(캐릭터) + WCL(성과). Firebase Auth + Functions OAuth 브로커(커스텀 토큰). 기존 한길련 회원은 "닉네임+PIN 입력 → 새 계정 병합" 1회 브릿지 후 PIN 폐지(브루트포스 취약점 아키텍처 해소).

### 6.2 캐릭터 등록
BNet 연동 → 계정의 캐릭터 목록 조회 → **만렙(90)만 노출** → 유저가 선택 등록(전부 등록 안 해도 됨) → **대표 캐릭터 지정**.

### 6.3 길드 가입 게이트 ★
- 가입 신청 시점: 등록 캐릭터 중 **해당 길드 소속 캐릭터가 1개 이상**이어야 신청 가능 (없으면 차단)
- 검증 방식: 유저 토큰은 24시간짜리 → 1회성 소유 확인에만 사용. **상시 검증은 공개 길드 로스터 API**(앱 자격증명, 유저 토큰 불필요)로 우리 서버가 주기 대조
- **전원 탈퇴 감지**: 일 1회 스케줄 함수가 길드 로스터와 플랫폼 명단을 diff → 등록 캐릭터가 로스터에 0개인 유저 발견 시 → 길드마스터·관리자에게 알림 **"○○님의 모든 캐릭터가 길드에서 확인되지 않습니다. 제명하시겠습니까?"** → [제명/유지/보류] 결정 (자동 제명 아님 — 벌점의 신고→관리자 결정 철학과 동일)
- 데이터 신선도: 블리자드 데이터는 로그아웃 시점 갱신(수 시간 지연 가능) → UI에 "마지막 확인 시각" + [지금 재확인] 버튼

### 6.4 커뮤니케이션
- **쪽지**: Firestore 대화방(당사자만 읽기), 차단·신고, 새 쪽지 디코 DM 알림(봇 활용)
- **게시판**: kgusystem 게시판 계승(공지/자유/구인) — 스코프별(플랫폼/연합/길드/공대)로 확장

## 7. 포인트 경제 — "꺼드럭거림"을 설계한다

### 7.1 적립 (출석 포인트)
1. 공대장이 출발 직전 로스터 **픽스**(최종 출발 멤버 잠금)
2. 일정 **종료 시각 경과** → 픽스 멤버에게 자동 지급 (+100P 기준)
3. 주 최대 **3회** (와우 주간 리셋 = 목요일 아침 기준 권장)
4. 미통보 노쇼: 공대장/일정 생성자가 **신고** → **플랫폼 관리자가 심사 후 벌점** (생성자는 벌점 권한 없음)
5. P4 애드온과 연동 시: 애드온으로 실제 초대된 명단 = 픽스 명단 자동 대조 → 출석 조작 원천 차단

### 7.2 수치는 3종 (과시 설계의 핵심)
- **잔액 P** — 상점용. 쓰면 줄어듦
- **누적 P** — 프로필 공개. 출석 총량의 증명 = 과시 지표
- **스트릭** — 연속 출석 주수

### 7.3 과시 장치
- **계급(티어)**: 누적 P 기준 와우식 계급 (예: 신병 → 정예병 → 백부장 → 천부장 → 전쟁군주) + 티어별 프로필 테두리
- **시즌 리더보드**: 전체 / 길드별 / 공대별 출석 랭킹 → 길드 대항전 효과
- **상점**: 칭호·프로필 테두리·배너 등 치장 전용. **현금 구매 절대 불가** → 착용 자체가 출석의 신호
- **시즌 한정판** 치장 + 시즌 종료 후 **명예의 전당**
- **프로필 카드 이미지 export**: 누적 P·계급·프로그레스가 박힌 카드를 디코에 자랑 → REKO의 바이럴 광고
- 원칙: 과시는 공개로, 망신은 비공개로 — **벌점 이력은 본인·관리자만 열람**. 신청 우선권 등 P2W 요소 금지

### 7.4 데이터
`pointLedger`(트랜잭션 원장: 지급/차감/벌점, 사유, 관련 raidId), 주간 캡 카운터, 시즌 집계 문서. 추가 아이디어(추후): 첫 킬 보너스, 용병 출석 보너스, 길드 단체 목표 보상.

## 8. AI 아트 파이프라인

UI는 AI 생성 이미지를 전제로 설계 — 프로토타입의 `ArtSlot` 컴포넌트가 규격 명시. 이미지는 대표님이 GPT로 생성.

| 슬롯 | 비율 | 용도 | 프롬프트 톤 가이드 |
|---|---|---|---|
| 히어로 키비주얼 | 16:9 | 랜딩 | 공허 궤도/포탈, 다크 바이올렛(#8A70FF 계열 발광), 미니멀 |
| 레이드 키아트 | 16:9 · 5:1 | 상세 헤더, 카드 | 공허첨탑 실루엣, 안개, 저채도+바이올렛 포인트 |
| 길드/공대 배너 | 4:1~6:1 | 프로필 상단 | 각 길드 정체성 (스타폴=별내림 숲 등) |
| 엠블럼 | 1:1 | 길드·공대 마크 | 문장(紋章)풍, 단색+바이올렛, 원/방패 실루엣 |
| 아바타/클래스 초상 | 1:1 | 유저·캐릭터 | 클래스 컬러 반영 일러스트 |
| 빈 상태 일러스트 | 1:1 | 일정 없음 등 | 쉬는 캐릭터 미니 일러스트, 유머 한 스푼 |

공통 프롬프트 접미어(안): "dark fantasy, deep black background #09090D, single violet accent #8A70FF, minimal composition, no text, high contrast rim light". 생성 후 `raidkorea/public/art/`에 규격별 폴더로 저장.

## 9. 기술 아키텍처 & 데이터 모델

- 스택: Vite + React 18 + Tailwind (프로토타입 그대로) + Firebase (Auth/Firestore/Functions) — kgusystem과 동일 계열이라 엔진 이식 마찰 최소
- `src/lib/mock.js`의 shape = Firestore 스키마 기준. 핵심 컬렉션:

```
users/{uid} · users/{uid}/identities/{provider} · users/{uid}/characters/{charId}
guilds/{guildId} · alliances/{allianceId} · teams/{teamId}
memberships/{uid}_{scopeType}_{scopeId}
raids/{raidId} (+apps/memos/cancels/logs — kgusystem 구조 계승, host 스코프 필드 추가)
messages/{threadId}/items · pointLedger/{txId} · progress/{scopeId} · seasons/{seasonId}
addonSessions/{code} (인게임 브릿지, TTL) · reports/{id} (노쇼 신고)
```

- 워커(Functions): BNet 로스터 감시(일 1회) · WCL 프로그레스 갱신 · 포인트 지급(종료 시각 트리거) · 디코봇(계승 확장)
- 인게임 애드온(P4): 로스터 임포트(코드 복붙) · 일괄 초대 · 귓말 초대 · 출석 export. 달력 초대는 12.x 보호함수 1일 검증 후
