## 2. 신설 — 공략게시판 (대표님 지시 260723)

### 2.1 콘셉트
레이드 던전의 **네임드별 공략 공유** 게시판. 익명 👍/👎 평가로 커뮤니티가 베스트 공략을 선별한다.

**전략적 의미**: 일정·로스터는 회원용이지만 공략은 **전 와우 유저가 원하는 콘텐츠**다. 비로그인 공개 + 검색 노출로 설계하면 "공허첨탑 3넴 공략" 검색 → 와니온 유입 → 회원 전환의 깔때기가 된다. 와니온의 첫 번째 SEO 자산.

### 2.2 구조
```
공략 (메뉴)
 └─ 던전 선택 (예: 공허첨탑 — 시즌 현행 던전이 기본)
     └─ 네임드 탭/목록 (1넴~8넴, 순서대로)
         └─ 해당 네임드의 공략 글 목록
             ├─ 🏆 베스트 공략 3 (도움 순추천 상위 3개 고정 노출)
             └─ 일반 목록 (기본: 등록순 / 정렬 토글: 도움순 · 최신순)
```

### 2.3 규칙
- **작성**: 로그인 유저 누구나. 제목+본문(+난이도 태그: 일반/영웅/신화, +선택: 영상/WCL 링크)
- **평가**: 글마다 [도움이 되었어요 👍 / 도움이 안 돼요 👎] — **표시는 완전 익명**(누가 눌렀는지 아무에게도 안 보임). 내부적으로는 1인 1표(변경·취소 가능), 본인 글엔 투표 불가
- **베스트 선정**: 순추천(👍−👎) 상위 3개, 동점 시 👍 많은 순 → 목록 최상단 "BEST" 섹션 고정
- **정렬**: 기본 등록순(오래된 순 아님 — 등록 최신순), 토글로 도움순/시간순
- **삭제 권한**: 기존 게시판 규칙 계승 (본인/플랫폼 운영자)
- **비로그인**: 열람 가능(투표·작성은 로그인)

### 2.4 데이터 모델
```
zones/{zoneId}                    — 던전 (이름, 시즌, 활성 여부)
zones/{zoneId}/bosses/{bossId}    — 네임드 (이름, 순서)  ← 관리자 게임데이터 도구로 시드
guides/{guideId}                  — { zoneId, bossId, difficulty, title, body, links[],
                                      authorId, authorName(스냅샷), createdAt,
                                      up, down, score(=up-down, 비정규화) }
guides/{guideId}/votes/{uid}      — { value: 1 | -1 }   ← 익명 표시용: 조회 규칙상 본인 것만 읽기 가능
```
- up/down 카운터는 투표 트랜잭션에서 ±1만 허용 (kgusystem commentCount 규칙과 동일 패턴)
- 네임드 목록은 공대 프로필의 킬 타임라인과 같은 zones/bosses 소스를 공유 (단일 소스)

### 2.5 확장 여지 (P3~)
- 와니온봇 `/공략 <네임드>` → 베스트 공략 링크 카드 응답
- 베스트 공략 유지 보상: 시즌 종료 시 포인트/칭호 「전술가」 (과시 경제 연결)
- 시즌 전환 시 이전 던전 공략 자동 아카이브

## 3. 포인트 경제 확정 스펙 (260723 대표님 지시 — P1에서 스키마 선반영, 지급 자동화는 P2~3)

**대원칙: 포인트는 계정 단위** (캐릭터가 아니라 유저에게 적립).

### 3.1 적립원 4종

| # | 적립원 | 규칙 | 기본값 제안* |
|---|---|---|---|
| ① | 레이드 출석 | 픽스 멤버로 출발 + 종료시각 경과 → 자동 지급. **주 3회 상한** (와우 리셋 목 기준) | +100P |
| ② | 일일 출석 체크 | 매일 1회, 사이트에서 [출석] 버튼 클릭 — **하루 경계는 한국시간 새벽 2시 리셋** | +10P |
| ③ | 공대장 완주 보너스 | 내가 공대장인 공대를 픽스하고 종료까지 완주 → 출석포인트에 **추가** 지급 | +50P |
| ④ | 공략 추천 보상 | 공략글 순추천(👍−👎) 마일스톤 도달 시 지급 (예: +10 / +30 / +50 도달) — 마일스톤당 1회만 | 100/200/300P |

*수치는 제안값 — 대표님 확정 필요. 마일스톤 방식인 이유: "추천 1개당 지급"은 부계정 어뷰징에 취약, 구간 돌파 보상은 안전.

### 3.2 공략 페널티 (어뷰징·저질 공략 방어)
- **비추천 > 추천(순추천 음수)인 공략글이 3개 이상 누적**된 유저 → `공략 보상 일시중지` 상태
- 중지되는 건 ④ 공략 추천 보상뿐 (글 작성·다른 적립원은 정상)
- 해제: 해당 글 삭제·개선으로 음수 글이 3개 미만이 되면 자동 해제 + 플랫폼 운영자 수동 해제 가능
- 상태는 본인·운영자만 열람 (벌점 비공개 원칙과 동일)

### 3.3 스키마 선반영 (P1-1에 포함)

users/{uid}: points{ balance, lifetime, streakWeeks, guideRewardSuspended }
pointLedger/{txId}: { uid, type: attend|daily|leader|guide|penalty|shop, amount, refId, season, at }
dailyCheckins/{uid_YYYYMMDD}         — 일일 출석 (중복 방지 키)
weeklyAttendance/{uid_weekKey}: { count }   — 주 3회 캡 카운터

## 4. Battle.net 필수 정책 확정 (260723)

| 정책 | 내용 |
|---|---|
| 연동 강제 | **BNet 미연동 유저는 레이드 신청 자체 불가** (하드 게이트). 열람은 가능 |
| 캐릭터 등록 | 연동 시 **만렙(90) 캐릭터 전원 자동 등록** (선택제 아님 — 변경 확정) |

| 작성자 표기 | 게시글·공략글·댓글의 작성자 = **대표 캐릭터명, 해당 클래스컬러로 표기** (작성 시점 스냅샷 — 비정규화 철학 계승) |
| 파생 효과 | 공략 작성·투표도 사실상 연동 필수 (작성자 표기에 대표캐릭 필요) — 미연동자는 '구경만' 가능한 계정 |

구현 노트: `users.mainCharId` 필수 필드, 신청·작성 진입점에서 `bnetLinked && mainCharId` 게이트. 대표캐릭 변경 시 과거 글 표기는 스냅샷 유지(변경 안 됨).

## 5. ★ 핵심 자산 5종 — 완전 구현 보증 (대표님 특별 지시 260723)

아래 5종은 한길련에서 심혈을 기울여 만든 와니온의 심장부다. **축소·간소화 없이 완전 이식**하며, P1 완료 판정의 필수 조건으로 삼는다.

| # | 자산 | 이식 기준 | 애드온 연동 준비 (P1에서 미리) |
|---|---|---|---|
| 1 | **공격대 시너지 트래커** | 버프·디버프 커버리지 판정 로직(raidAbilities) 그대로 + 레이드 상세에 전면 배치 | — |
| 2 | **스왑 가능자 시스템** | 다중 특성 기반 역할 전환 표시 + 스왑 가능자 목록 완전 재현 | 스냅샷 코드에 스왑 정보 포함 |
| 3 | **대기·벤치 시스템** | 정원 초과 자동 대기, 벤치 신청, 승격/강등 플로우 그대로 (race만 트랜잭션 보완) | 벤치 호출 명령(/wanion 벤치)용 데이터 포함 |
| 4 | **시뮬레이션 (파티 배치)** | 드래그 배치 + 프리셋 완전 이식 (자동저장은 분리) | **[인게임 배치 코드 복사] 버튼 P1 구현** → `WANION1;` 포맷에 파티번호 포함 → P4 애드온 `/wanion sort`가 소비 |
| 5 | **구성원 초대** | 확정 로스터 관리 UI 그대로 | **[인게임 초대 코드 복사] 버튼 P1 구현** → 픽스 명단 코드 생성 → P4 애드온 `/wanion invite`가 소비 |

원칙: 웹 쪽 브릿지(코드 생성·파싱 유틸, 버튼 UI)는 **P1에서 완성**한다. 애드온(P4)이 나오는 날, 웹은 이미 준비돼 있어서 애드온만 꽂으면 바로 동작한다.

## 7. v1.2 확정 — 손님파티 · 프로필카드 · 정공 리포트 · 수락 모드 · 과거 열람 (260723 대표님 지시)

### 7.1 손님파티 시스템
- **레이드 유형 플래그**: 파티 유형을 '손님파티'로 지정해야만 손님 관련 UI가 나타남. 일반 파티엔 손님 요소 0
- **전환 규칙**: 일반→손님파티 전환 자유. 손님파티→일반 전환은 손님이 1명이라도 있으면 차단 + 경고("손님을 제거하거나 일반 공대원으로 변경해주세요"). ⇒ '손님→일반 공대원 변경' 기능 필요
- **관리 권한**: 손님파티는 **생성자 + 슈퍼관리자만** 관리 (스코프 관리자 아님)
- **업비(골드)**: 생성자만 열람·수정. 기본 비공개. 생성자가 [업비 공개] 실행 시 **경고창 2회(더블 컨펌)** 후 공개 — 공개 범위는 **그 공대에 소속된 실제 공대원만** (손님·비신청자·외부인 절대 불가)
- **정원**: 손님도 정원 포함. 표기 예: `20/20 (공대원 18 + 손님 2)` — 영웅 30인에 손님 10+선수 20 같은 대량 버스 구성도 가능
- **손님 정보**: 캐릭명+서버 필수(달력·초대 코드용), **클래스 필수**(시너지 뉘앙스 직결). 손님유형(깡/업 등)·업비는 선택 입력
- **등록 주체**: 생성자·슈퍼관리자 수기 등록 + BNet 연동 회원의 '손님 지원'(생성자/슈퍼관리자 승인 후 확정)
- **시너지**: 손님은 시너지 카운팅 제외, 미충족 항목에 "손님 중 전사 1" 수준의 힌트 배지 (확정)
- **포인트**: BNet 연동 회원 손님은 출석포인트 동일 지급
- **초대·달력**: 손님도 WANION1 초대/배치 코드 및 달력 초대 명단에 포함 (핵심 목적)

### 7.2 프로필 카드 (v1.2 갱신)
- 공격대 로고: **복수 소속 시 복수 선택 가능** (길드 로고는 1개 선택 — 규칙 상이 주의)

## 8. v1.3 확정 — 스코프 게시판 3층 구조 + 페이지 접근 모델 (260723)

### 8.1 게시판 = 스코프 3층 × 2종
| 층 | 통합 게시판 | 공략 게시판 |
|---|---|---|
| 전체(플랫폼) | 공지/자유/구인구직/거래 | 시즌 던전 · 보스별 토글 |
| 길드별 (길드 페이지 내) | 동일 카테고리 | 동일 구조 |
| 공격대별 (공격대 페이지 내) | 동일 카테고리 | 동일 구조 |

### 8.2 공략 다중 공개 (크로스 포스팅)
- **글은 하나, 공개 범위만 체크박스**: [전체 공개] [길드에 공개] [공격대에 공개] 자유 조합 (1~3개)
- 수정 시 공개 범위 추가 가능 (길드에만 쓴 글 → 나중에 전체 공개 켜기)
- 복사본이 아니라 단일 문서의 다중 노출 → 추천수(👍👎) 공유, 베스트3는 각 스코프 목록 내에서 산출
- 데이터: guides.scopes = [{type:'global'} | {type:'guild', id} | {type:'team', id}] 배열 + array-contains 쿼리

### 8.3 손님파티 미결 4건 확정
1. 손님은 **탱힐딜 역할 정원에 불포함** — 총원에만 포함 ("20/20 (공대원 18+손님 2)")
2. 손님 유형 프리셋: 길드·공격대 **관리자는 자기 페이지에서 프리셋 관리** / 글로벌 공대장은 매번 수기 입력
3. 정공 레이드 수락: **정공원 자동수락 + 해당 일정만 본인 스스로 불참 처리 가능**(다음 일정엔 정규 로스터라 자동 복귀) / 외부 인원은 검토후수락
4. 손님파티 **보드 공개 노출 O** — 단 손님 유형·업비는 신청 전 비공개, 공대원이 되어도 공대장이 비공개면 안 보임

### 8.4 길드·공격대 페이지 접근 모델
- 네비: [길드] 탭 = 내 소속 길드 목록 / [공격대] 탭 = 내 소속 공격대 목록
- 페이지 뷰 분기:
  - **외부인**: 공개 프로필만 (소개글·프로그레스 — TeamSAD 소개 사이트 구성 참고: C:\VibeCoding Claude\TeamSAD)
  - **소속원**: **게시판이 디폴트 메인** (소개는 이동 가능하되 기본 아님)
  - **관리자**: 게시판 디폴트 + 관리 페이지 진입. 관리 페이지는 일반 소속원에게 존재 자체 비노출
- 게시판(통합·비공개 공략)은 소속원 전용. 외부인은 소개 뷰만

### 8.5 카테고리 정책 (확정)
- 통합 게시판 카테고리는 **관리형 목록** — 하드코딩 금지. 초기값: 공지/자유/구인구직/거래
- 전체 게시판 카테고리는 플랫폼 운영자가, 길드·공격대 게시판 카테고리는 각 관리자가 추가·삭제 (코드 수정 없이)
- 규칙(rules)도 고정 enum 검증 대신 카테고리 문자열 + 공지 권한류만 검증하는 방식으로 완화
